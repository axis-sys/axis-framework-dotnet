# CLAUDE.md

Instructions for an AI agent (or human) working in this repo — the Axis .NET framework, its canonical
rule base (`rules/`), its reference application (`src/scaffolds/`) and the Claude Code plugin it ships
(`skills/`).

## Never guess a concrete name or path — check `src/scaffolds/` first

`src/scaffolds/` is the compiled, tested sample solution and the ground truth for every convention
`rules/conventions/` describes. When a rule, skill or doc needs a concrete project name, folder nesting,
namespace or class shape and you don't have it verified: **grep the scaffold for the real one before
writing the sentence.** The naming canon at a glance (five sibling solution folders):

- `Core/` → `{App}.SharedKernel`, `{App}.Domain`, `{App}.Application`
- `Contracts/` → `{App}.Contracts.Driving`, `{App}.Contracts.Driven` — **sibling of `Core/`, never nested**
- `Adapters/` → `{App}.Adapters.Driving.Facade` (facade impls live here, never in the Application),
  `{App}.Adapters.Driven.{Role}`
- `Host/` → `{App}.Host` (the one composition root)
- `Tests/` → `{App}.{Kind}Tests` (Unit, Integration, E2E, Architecture)

## A rule is not done without a concrete anchor

For a `must`-severity rule under `rules/conventions/{architecture,edge,testing}/`, `reference.kind:
none-yet` with empty `examples` is a lint **error** (ratcheted; the grandfathered debt list in
`rules/tooling/lint-rules.mjs` only shrinks). Point `reference.at` / `examples.correct` at real
`path:line` anchors in `src/scaffolds/` (kind `framework-sample`) or framework source/tests. A rule that
names a thing only by role ("the driving contracts project") with no anchor is exactly how a downstream
agent ends up guessing a name that compiles and is wrong.

`derives_to.scaffolds` entries must match a real `#region scaffold:<id>` marker under
`src/scaffolds/**/*.cs` (lint-enforced). Leave the list empty rather than inventing a name.

## The plugin ships a bundle — keep it in sync

The installable plugin subtree (`skills/plugins/axis-dotnet/`) carries a generated copy of `rules/`,
`docs/` and `src/scaffolds/` (see its `BUNDLE.md`), because deep-relative links out of the subtree are
dead in every real installation. After touching rules, docs, scaffold sources or SKILL.md links, run
`node skills/scripts/bundle-plugin.mjs` (write mode) — CI fails on drift. Never edit the bundled copies
by hand.

## The private reference application

Patterns may be *extracted* from the private reference app, but **no reference to it — name, domain,
namespace or path — may appear anywhere in this public repo**. Generalize every pattern into the
ECommerce scaffold domain before it lands here.

## Checks to run before handing work back

```
dotnet build src/scaffolds/Scaffolds.slnx -c Release -warnaserror   # scaffold builds clean
dotnet test  src/scaffolds/Scaffolds.slnx -c Release                # architecture/unit/E2E stay green
node rules/tooling/lint-rules.mjs                                   # schema + anchors + ratchet
node rules/tooling/check-enforcement.mjs                            # rule ↔ analyzer links
node rules/tooling/sync-scaffolds.mjs --check                       # doc snippets match the scaffold
node rules/tooling/check-doc-parity.mjs                             # pt-br docs mirror en-us structure
node skills/scripts/bundle-plugin.mjs --check                       # plugin bundle is current
node skills/scripts/lint-skills.mjs                                 # skill maps well-formed, no dead links
```

See `rules/rules-guide.md` for the full rule-authoring workflow and `rules/_schema.md` for the schema.
Never hand-edit `rules/README.md` — it is generated by `gen-index.mjs`.
